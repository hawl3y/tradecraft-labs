**Release date:** 2026-06-24

## Release Summary

v0.1.0 is the foundational release of Tradecraft Labs — an independent research platform for cybersecurity governance, leadership, and risk-informed decision making. The platform is built to publish research in progress: working theories, raw observations, literature frameworks, and evolving analysis, not only final papers.

This release establishes the complete platform infrastructure, the full content architecture for Project 001 (BISO as Governance Intermediary), and a production deployment baseline. All routes are functional, the build is statically generated, and the platform is ready for public deployment.

## Research Capabilities

Tradecraft Labs is built to serve the research process, not just publish its outputs. At v0.1.0, the platform enables:

**Documenting work in progress**  
Research artifacts publish at any stage of completion. A working theory, a literature framework, or a raw observation can be published as an artifact the moment it has value — not only after it is finished. Stub artifacts hold the place and preserve the URL before content is ready.

**Credibility in interview outreach**  
The platform serves as a research evidence artifact when approaching potential interview participants. A link to the research question, working thesis, and published observations demonstrates that the research is structured and serious. Visitors can read the interview analysis framework before agreeing to participate.

**Research stage transparency**  
The seven-stage research timeline on the homepage and project pages shows where the research currently stands. At launch: Interviews (Stage 4 of 7). Visitors understand this is active research in progress, not a completed body of work.

**Iterative thesis development**  
The working thesis is versioned (`v2` in the project frontmatter) and labeled as working. As interviews complete and analysis develops, the thesis will be updated in place. The journal documents the reasoning behind each revision.

**Paper publication pathway**  
The Publications page, `PublicationCard` component, and `Publication` Contentlayer type are fully implemented and ready. When a paper is submitted or published, adding it requires one MDX file and a PDF in `public/downloads/`.

**Future project expansion**  
All components, routes, and content types handle multiple projects automatically. Adding Project 002 requires only new MDX files in `content/` — no code changes.

## Architecture Summary

### Route Map (24 routes)

```
Static (○):
  /                          Homepage
  /about                     About
  /publications              Publications (empty state)
  /research                  Research listing
  /opengraph-image           OG image (generated)
  /robots.txt                Robots (generated)
  /sitemap.xml               Sitemap (generated)
  /_not-found                404

SSG (●) — prerendered via generateStaticParams:
  /research/[project]                          1 route
  /research/[project]/artifacts                1 route
  /research/[project]/artifacts/[artifact]     9 routes
  /research/[project]/journal                  1 route
  /journal/[entry]                             1 route

Dynamic (ƒ):
  /journal                   Server-rendered (searchParams filtering)
```

### Content pipeline

```
/content/
  projects/       MDX → Project type → /research/[slug]
  artifacts/      MDX → Artifact type → /research/[project]/artifacts/[slug]
  journal/        MDX → JournalEntry type → /journal/[slug]
  publications/   MDX → Publication type → /publications (future)
```

### Key design decisions

- **Date handling:** All date strings parsed with `'T00:00:00'` suffix to prevent UTC midnight off-by-one in US timezones
- **Stub artifacts:** `status: "stub"` renders "Content pending" instead of sparse MDX body — all routes resolve
- **Server-props filter pattern:** Journal uses server-side `searchParams` passed as props to client filter component, avoiding `useSearchParams` Suspense requirement
- **CSS Grid timeline:** `repeat(7, minmax(0, 1fr))` ensures stage labels align under dots without coordinate math
- **OG image:** Uses system Georgia serif — Satori (ImageResponse engine) requires TTF/OTF; Google Fonts CDN serves WOFF/WOFF2 which Satori rejects

## Recommended Git Workflow

### Branch Strategy

```
main          Production branch — Vercel auto-deploys on push
              Always deployable. Only push when build passes locally.

dev           Active development
              Use for any code changes. Merge to main when ready.

content/*     Content-only branches (new journal entries, artifact MDX)
              Can push directly to main — no code changes, no risk.
              Example: content/journal-interview-round-1

feature/*     Feature development (new components, infrastructure)
              Merge to dev first, then dev to main.
```

For a solo researcher/developer, the rule is simple: **if `npm run build` passes locally, it's safe to push to main.** Use branches as a safety net, not a ceremony.

### Commit Strategy

Short, descriptive messages in present tense. No tooling required.

```
<type>: <description>
```

Types:
```
content:    New or updated MDX (journal entries, artifacts, project files)
feat:       New platform feature or component
fix:        Bug or visual correction
docs:       Documentation changes (release notes, review docs)
config:     Configuration changes (Next.js, Tailwind, Contentlayer)
chore:      Maintenance (dependencies, .gitignore, cleanup)
```

Examples:
```
content: add journal entry for first interview round
content: convert research proposal artifact to MDX
feat: add citation copy-to-clipboard to CitationBlock
fix: correct date parsing for UTC offset in sitemap
docs: add v0.1.0 release notes and push readiness checklist
chore: exclude Interview_Pool from git tracking
```

### Tagging Strategy

Tags mark meaningful milestones in the research platform. Use annotated tags.

```
v0.MAJOR.MINOR

v0.x.0  — milestone releases (phases, significant content additions)
v0.x.y  — patch releases (bug fixes, content updates, copy changes)
v1.0.0  — first paper published / platform fully content-complete
```

Tag this release:
```bash
git tag -a v0.1.0 -m "Initial platform release — Tradecraft Labs"
git push origin v0.1.0
```

Suggested milestone tags:

| Version | Trigger |
|---|---|
| `v0.1.0` | This release — platform infrastructure complete |
| `v0.2.0` | Phase 7 complete — all artifacts converted to MDX |
| `v0.3.0` | First paper submitted for publication |
| `v1.0.0` | First paper published |

### Release Strategy

A release captures a meaningful state of the platform — not every commit.

**When to cut a release:**
- Platform infrastructure milestone (this release)
- All artifacts converted and published (Phase 7 complete)
- First paper added to Publications
- Major new capability shipped (search, RSS, etc.)

**Release workflow:**
1. `npm run build` locally — confirm 0 errors
2. Update or create `docs/releases/vX.Y.Z.md`
3. Merge final changes to `main`
4. Push to GitHub — Vercel deploys automatically
5. Create and push annotated tag
6. Run `docs/reviews/Production-Validation-Checklist.md`
